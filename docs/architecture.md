# Architecture

L-UI is split into a hub role and a node (agent) role.

## Hub

- Runs the main panel, API, and cron jobs
- Owns users, nodes, inbounds, clients, routing, and assignments
- Renders the dashboard, charts, and node overview
- Builds and ships node bundles during bootstrap
- Talks to agents through signed HTTP API calls after bootstrap completes
- Never runs Xray itself — it is a pure control plane

## Agent

- Runs a lightweight standalone binary on each VPS
- Keeps runtime files, systemd units, and Xray config locally
- Stores its own JWT secret for hub↔agent authentication in a local SQLite database
- Starts and manages Xray under systemd
- Exposes status, logs, metrics, config, firewall, and lifecycle endpoints
- Supports TLS listener for encrypted hub↔agent communication

## Data Flow

1. The hub stores configuration and node metadata in its database.
2. The panel renders the data and refreshes node status over HTTP.
3. The hub bootstraps nodes over SSH and SCP using bundled artifacts.
4. The node agent reports heartbeats back to the hub over signed HTTP.
5. The hub shows stale data explicitly when the heartbeat stops.
6. The hub pushes configuration changes (Xray config, client lists, firewall rules, TLS certs) to the agent on demand.
7. The agent applies changes locally and reports results.

## Frontend Split

- The frontend is a multi-page Vite app under `frontend/`, not a single SPA bundle
- Each top-level page has its own HTML entry and bootstrap file: `index.html` (panel), `login.html`, `subpage.html`
- `hub/web/dist/` is generated by Vite and embedded into the Go binary
- The Go binary serves embedded assets in release mode and disk assets in debug mode

## Startup Order

The normal operator flow is:

1. Start the Go backend.
2. Wait for `/csrf-token` to respond.
3. Start the frontend.
4. Open the panel in a browser.

`make dev` follows this order directly.

## Bootstrap

1. Admin enters SSH credentials in the panel.
2. Hub validates the request and starts an async bootstrap job.
3. The goroutine connects over SSH and detects the remote architecture.
4. Hub builds and uploads a node bundle over SCP (with retry on failure).
5. Hub installs systemd units and starts the node agent.
6. Hub switches to API control and polls for liveness.
7. The node appears in the panel with status, metrics, and config version.

Bootstrap uses retry logic at every network-dependent step:
- SSH dial: 3 attempts with exponential backoff (2–10s)
- Bundle upload: 3 attempts with 1–3s backoff
- Bundle download (on the hub side): 3 attempts with 1–5s backoff
- Agent verification: polls with randomized jitter (2–4s) to reduce thundering herd

## Node Lifecycle

1. **Registered** — node is created in the database (via bootstrap or registration token).
2. **Online** — node heartbeat succeeds; status, metrics, and version are visible.
3. **Config pushed** — Xray config and client list are sent to the node.
4. **Drift detected** — the hub compares the stored config version and shows drift reasons.
5. **Offline** — heartbeats stop; the hub alerts via Telegram after the configured down threshold.
6. **Reconciled** — the hub attempts to probe, restart Xray, or reinstall the bundle to restore the node.
7. **Deleted** — the hub removes the node and optionally runs cleanup on the agent.

## Monitoring

- The `NodeHeartbeatJob` runs every 5 seconds, calling `Probe()` on each enabled node.
- `Probe()` uses retry logic (3 attempts, 200ms–2s backoff) for transient HTTP failures.
- On success, `UpdateHeartbeat()` records status, latency, CPU, MEM, disk, network traffic, and versions.
- The `NodeAlertJob` runs every 10 seconds, checking consecutive offline count and resource thresholds.
- Alerts are sent to configured Telegram chat IDs.

## TLS Certificate Flow

1. Hub generates a CA (certificate authority) and stores it in settings.
2. Hub generates a node certificate signed by the CA on demand.
3. Hub pushes the certificate and key to the agent via `POST /api/v1/certs`.
4. Agent stores the cert and restarts its HTTP listener with TLS.
5. The `NodeCertRenewalJob` runs daily, checking for certificates expiring within 30 days and renewing them.

## Retry Architecture

Transient failures in hub↔agent communication are handled by the `internal/util/retry` package:

- **Hub → agent HTTP calls**: All idempotent `Remote` methods use `doWithRetry()` with 3 attempts and 200ms–5s exponential backoff.
- **Non-idempotent operations** (AddInbound, PushConfig, InstallXray, etc.) use a single attempt to avoid duplicate side effects.
- **Transient detection** covers connection refused, DNS failure, TLS handshake errors, timeouts, resets, broken pipes, EOF, 5xx, and 429 responses.
